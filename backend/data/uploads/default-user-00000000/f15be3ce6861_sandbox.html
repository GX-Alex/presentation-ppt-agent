一、项目概述

了解 TransactionOCR_2 的核心定位与技术价值

📋 项目简介

TransactionOCR_2 是一个专业的银行交易流水 OCR 智能体服务，核心功能是将 PDF 或图片格式的银行流水账单通过视觉语言模型（VLM）自动识别提取，并转换为结构化的标准 CSV 数据。

该项目采用 FastAPI + LangGraph 的现代微服务架构，通过阿里云百炼 Qwen 系列模型实现高精度的多模态信息提取，同时支持 Dify 平台的动态提示词管理，具备企业级生产部署能力。

FastAPI LangGraph Qwen-VL PyMuPDF Pandas Dify

🎯 核心能力

  • 多格式支持 — 支持 PDF、图片（PNG/JPEG）等多种银行流水文档格式
  • 智能识别 — 基于视觉语言模型，自动理解复杂表格结构和多银行格式
  • 字段标准化 — 自动映射到 23 个标准字段（流水号、交易日期、对方账户、金额等）
  • 余额校验 — 内置余额连续性校验，自动修正借贷方向错误
  • 异步处理 — 支持大文件分块并发处理，完成后回调通知
  • 动态提示词 — 支持 Dify 平台热更新提示词，无需重新部署

    三、核心功能模块详解

    四大核心模块协同实现智能流水识别

    🔄 OCR Agent 工作流（LangGraph）

    采用经典的 Fan-out/Fan-in 并行模式，两个 LLM 节点同时调用，充分利用模型推理时间重叠：

    # LangGraph 工作流结构
START
  ├── LLM 1: extract_transactions  →  从图片提取交易流水 Markdown 表格
  └── LLM 2: extract_metadata      →  从表头区域提取客户元数据 JSON
         ↓
      merge_results                →  合并流水与元数据为最终 JSON
         ↓
       END
    ✓ 支持多图批量处理
    ✓ 表头感知（非首块携带第一页参考）
    ✓ 元数据去重（分块场景下非首块跳过）

    📑 PDF 分块并发处理

    当 PDF 页数超过 CHUNK_SIZE（默认 8 页）时，系统自动：

  • 智能分块 — 按 CHUNK_SIZE 将页面分组，非首块前插入第一页作为表头参考
  • 并发控制 — MAX_CHUNK_WORKERS（默认 6）限制同时发往 LLM 的块数
  • 指纹去重 — 基于 (trans_date, trans_time, trans_amt, account_balance) 去除重叠行
  • 元数据传播 — 从 chunk-0 提取的 name/account_no/account_bank 填充其他块的空值

    • 重试与二分策略

  • 每次失败自动重试 2 次（可配置）
  • 行数过少时触发系统性截断检测
  • 自动二分拆小后递归处理（最多 2 层）

    • ⚖️ 余额连续性校验（Balance Corrector）

    这是本项目最具技术深度的模块之一，通过财务逻辑一致性校验自动修正 OCR 错误：

    📈 升序校验

    时间从早到晚：

    bal[i] = bal[i-1] + delta[i]
# delta[i] = +amt（贷方） 或 -amt（借方）

    📉 降序校验

    时间从晚到早：

    bal[i] = bal[i-1] - delta[i-1]
# 前一行的借贷方向决定变化

    修正逻辑

  • 余额差在容忍值内（默认 0.01 元）→ 标记"正确"
  • 翻转借贷方向后吻合 → 自动修正并标记"已修正"
  • 无法修正 → 标记"存疑"供人工复核

    • 🗺️ 字段映射与标准化

    transaction_mapper.py 实现了完整的字段标准化流程，支持 23 个标准字段：

    目标字段清单

    字段名 说明
    id, serial_id序号标识
    name, account_id, account_no账户信息
    account_bank开户机构
    trans_date, trans_time交易时间
    opponent_name对方账户名
    debit_credit借贷方向
    trans_amt, account_balance金额与余额
    currency币种
    trans_channel, trans_type渠道与类型
    trans_use, abstract, remark用途与备注

    映射策略

  • 精确匹配字段名（如"对方户名"→"opponent_name"）
  • 模糊匹配包含关系（如含"余额"→"account_balance"）
  • 支持中英文别名（如"户名"→"name"，"支出"→"trans_amt"）
  • 日期时间格式自动标准化（YYYY-MM-DD、HH:MM:SS）

    RESTful 风格 API，开箱即用的集成体验

    核心 OCR 接口

    接口 方法 说明
    /ocr POST 异步受理，接收 URL 列表，立即返回 202，后台处理完成后回调
    /ocr/upload POST 同步处理，本地上传文件测试用，直接返回 CSV 结果

    字段映射服务

    接口 方法 说明
    /map POST CSV 字段映射，返回标准化 CSV
    /map/json POST 返回 JSON 格式预览（限 100 条）
    /map/info POST 分析映射关系

    五、技术亮点与创新

    企业级可靠性与高并发优化设计

    🏢 企业级可靠性设计

  • 全局异常处理 — 统一捕获未处理异常，返回标准错误响应
  • 超时控制 — LLM 请求 1200 秒超时，HTTP 回调 60 秒
  • 重试机制 — 指数退避重试（1s→3s→5s），避免雪崩
  • 回调保活 — 连接复用关闭，防止 F5 断连
  • 原始报文日志 — RawLoggingMiddleware 记录上下游请求/响应
  • Base64 掩码 — 自动截断 base64 data URL，避免日志暴增

    • ⚡ 高并发优化

  • 异步架构 — 全程 async/await，无阻塞 IO
  • 信号量控制 — MAX_CHUNK_WORKERS 限制 LLM 并发压力
  • PDF 分块 — 大文件自动拆分并发处理
  • 多文件并发 — PDF 与图片文件并行处理

    • 🔒 数据质量保障

  • 余额连续性校验 — 基于财务规则的自动纠错
  • 行数阈值检测 — 防止 API 截断导致数据丢失
  • 指纹去重 — 多 chunk 合并时消除重复行
  • 日期时间校验 — 格式 + 日历合法性双重校验

    • 📊 可观测性

  • 结构化日志 — 带时间戳的 DEBUG/INFO/WARNING/ERROR 分级日志
  • 缓存状态监控 — cache_status() API 暴露缓存健康状态
  • TTL 缓存 — 提示词缓存 10 分钟，支持手动失效

    • 六、配置灵活性

    丰富的环境变量配置，支持企业级定制

    核心配置项

    配置项 说明
    LLM 配置
    LLM_API_KEY, LLM_BASE_URL, LLM_MODEL 阿里云百炼 API 凭证与模型选择
    LLM_ENABLE_THINKING, LLM_EXTRACT_TEMPERATURE 思考模式开关与温度参数
    并发控制
    MAX_WORKERS, CHUNK_SIZE, MAX_CHUNK_WORKERS 并发数、分块大小、LLM 并发限制
    PDF 处理
    PDF_ZOOM, PDF_QUALITY, PDF_MAX_WIDTH, IMAGE_FORMAT 缩放倍数、图片质量、最大宽度、输出格式
    业务参数
    BALANCE_CORRECTION_TOLERANCE 余额校验容忍值
    CALLBACK_URL, DIFY_BASE_URL 上游回调地址、Dify 平台地址
    所有配置均通过 .env 文件管理，支持本地开发、测试环境、生产环境的差异化配置，无需修改代码即可切换部署环境。
