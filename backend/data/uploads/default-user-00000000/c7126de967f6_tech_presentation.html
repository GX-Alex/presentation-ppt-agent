01. LangChain 生态与演进

LangChain Ecosystem & Evolution

LangChain 是什么？

LangChain 是一个用于开发 LLM 驱动应用的开源框架。

核心价值

• 统一的 LLM 接口抽象 (OpenAI, Claude, 本地模型...)
• 丰富的组件库 (Prompt, Chain, Memory, Tool...)
• 简化 LLM 应用开发流程

GitHub Star 90k+，LLM 应用开发事实标准

生态全景

langchain-core核心抽象与接口
langchain组件库 (Chain, Agent)
langchain-community第三方集成
langgraph图状态机引擎
langserveAPI 服务部署
langsmith可观测性平台

LangChain 核心组件

Chat Models

统一接口调用 GPT-4, Claude, 通义千问等各种 LLM

Prompt Templates

可复用、可组合的提示词模板系统

Chains (LCEL)

用管道操作符 | 组合多个组件

Retrievers

RAG 检索器，支持向量库、BM25 等

Tools

可被 LLM 调用的外部功能封装

Output Parsers

将 LLM 输出解析为结构化数据

LCEL：LangChain Expression Language

LCEL 是 LangChain 的声明式组合语法：

  • | 管道符：前一步输出作为后一步输入
  • Runnable：所有组件的统一接口
  • invoke/stream/batch：统一调用方式
局限性：LCEL 只能构建 DAG（有向无环图），无法实现循环！ 

from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI
from langchain_core.output_parsers import StrOutputParser

# LCEL: 用 | 组合组件
prompt = ChatPromptTemplate.from_template(
    "用中文解释：{concept}"
)
model = ChatOpenAI(model="gpt-4o")
parser = StrOutputParser()

# 构建 Chain
chain = prompt | model | parser

# 统一调用接口
result = chain.invoke({"concept": "量子计算"})

LLM 应用架构的三个阶段

1. Chain (链)

线性流水线
Prompt -> LLM -> Parser

2. Router (路由)

条件分支
If X then A else B

3. Agent (循环)

Graph 特性
Think -> Act -> Observe -> Loop

Agent 的本质公式

Agent = LLM + Memory + Planning + Tools

LangGraph 为这三个核心能力提供了原生支持。

Chain vs Graph：核心差异

维度 LangChain (LCEL) LangGraph
拓扑结构 DAG (有向无环图) Cyclic Graph (允许循环)
状态管理 隐式传递 * 显式状态机 *
执行模型 一次性流水线 迭代式执行 (可循环)
持久化 需手动实现 内置 Checkpointer

* 隐式传递 (Pass-through)：数据像水流一样从上游自动流向下游，无需手动管理状态，但无法在中途修改或持久化。

* 显式状态机 (Centralized State)：所有节点共享一个中央 State 对象，节点读取状态、返回更新，由运行时统一合并管理。

何时选择 LangGraph？

LangChain LCEL 足够

• 简单的 RAG 问答
• 一次性数据提取/转换
• 线性工作流 (A→B→C)
• 不需要记忆的单轮对话

需要 LangGraph

• Agent：LLM 决定下一步
• 循环：失败重试、迭代优化
• 持久化：多轮对话、断点续传
• 人机协同：审批、确认
• 多智能体协作

LangGraph 架构全景图

架构说明：定义层 (StateGraph) → 执行层 (Agent/Tool/State) → 持久化层 (Checkpointer/Store)；外部服务通过 Tool 接入

Core Classes & Architecture

LangGraph 核心类关系

蓝图 vs 可执行体。类比：Class 定义 vs 实例化对象

2.1 State：中央数据结构

State 是图中所有节点共享的数据容器，采用 TypedDict 定义结构。

数据中心

节点间唯一通信渠道，读取输入、写入输出

执行上下文

携带对话历史、中间结果、控制变量

持久化载体

Checkpointer 序列化 State 实现断点续传

路由依据

条件边根据 State 决定下一节点

Reducer 合并策略： add 追加列表 | 默认 覆盖旧值 | 自定义函数 任意合并逻辑 

from typing import TypedDict, Annotated
from operator import add
from langgraph.graph import MessagesState

# 方式1: 自定义 State
class AgentState(TypedDict):
    messages: Annotated[list, add]  # 追加模式
    context: str                     # 覆盖模式
    iteration: int                   # 覆盖模式

# 方式2: 继承预定义 MessagesState
class MyState(MessagesState):
    # 已包含 messages: Annotated[list, add]
    custom_field: str

# 自定义 Reducer 示例
def merge_dicts(old, new):
    return {**old, **new}

# Node 中如何使用 State
def my_node(state: AgentState):
    # 读取 State
    msgs = state['messages']
    # 返回更新（由 Reducer 自动合并）
    return {"messages": [AIMessage("...")]}

2.2 Node：执行单元

节点是标准 Python 函数，设计理念借鉴了 Actor 模型。

Actor 模型理念：Node 读取 State 但不直接修改，而是返回更新消息。这种设计使得 LangGraph 可以：
• 安全地进行状态合并 (通过 Reducer)
• 支持 Checkpointing (每步状态可序列化)
• 方便调试和回溯 (输入输出明确)
关于并发：Node 本身是顺序执行的，由 Edge 控制流转。并发只发生在特定场景：
• Send API: 使用 Send() 显式创建并行分支
• ToolNode: 内部可并行执行多个工具调用 

def agent_node(state: AgentState):
    """
    Node 的黄金法则：
    1. 读取 State (只读快照)
    2. 执行业务逻辑 (调用 LLM/Tool)
    3. 返回 Update Dict (增量更新)
    """
    messages = state['messages']
    response = model.invoke(messages)
    # 返回增量，由 Reducer 合并到 State
    return {"messages": [response]}

执行顺序：由 Edge 定义，START → Node A → Node B → END

2.3 Edge：控制流

两种边类型

  • Static Edge: A -> B (固定路由)
  • Conditional Edge: 根据逻辑动态路由

# 静态边
workflow.add_edge("node_a", "node_b")

# 起点和终点
workflow.add_edge(START, "node_a")
workflow.add_edge("node_b", END)

条件边示例


def should_continue(state):
    last_msg = state['messages'][-1]
    if last_msg.tool_calls:
        return "tools"  # 有工具调用
    return END          # 结束

workflow.add_conditional_edges(
    "agent",           # 源节点
    should_continue,   # 路由函数
    ["tools", END]     # 可能的目标
)

2.4 CompiledGraph：可执行体


from langgraph.graph import StateGraph, START, END

# 1. 构建蓝图
workflow = StateGraph(AgentState)
workflow.add_node("agent", agent_node)
workflow.add_node("tools", tool_node)
workflow.add_edge(START, "agent")
workflow.add_conditional_edges("agent", should_continue)
workflow.add_edge("tools", "agent")

# 2. 编译为可执行图
app = workflow.compile()

# 3. 调用 (与 LangChain Runnable 完全兼容)
result = app.invoke({"messages": [HumanMessage("Hello")]})

分形架构：因为 app 是 Runnable，可将整个图作为一个 Node 嵌入另一个图 (Subgraph)。

03. 工具调用 (Tool Calling)

Agent 与外部世界交互的桥梁

工具调用全流程图

工具调用的核心是 LLM 决定"是否"以及"调用哪个"工具，而不是硬编码逻辑。

3.1 定义工具：@tool 装饰器

@tool 装饰器作用：
• 将普通 Python 函数转换为 LangChain Tool 对象
• 自动从函数签名提取参数类型和名称
• 自动从 Docstring 提取工具描述和参数说明
• 生成符合 OpenAI Tool Calling 规范的 JSON Schema

Docstring 是关键

LLM 通过 Docstring 理解工具用途，决定何时调用。描述要清晰、准确、包含使用场景。


from langchain_core.tools import tool

@tool
def multiply(a: int, b: int) -> int:
    """Multiply two numbers.
    
    Args:
        a: First integer
        b: Second integer
    """
    return a * b

@tool
def search_database(query: str) -> str:
    """Search the internal database for information.
    
    Args:
        query: The search query string
    """
    # 实际实现...
    return f"Results for: {query}"

3.2 绑定工具：bind_tools()


from langchain.chat_models import init_chat_model

# 初始化模型
model = init_chat_model("gpt-4o", temperature=0)

# 定义工具列表
tools = [multiply, search_database]

# 将工具绑定到模型
model_with_tools = model.bind_tools(tools)

# 现在模型可以决定是否调用工具
response = model_with_tools.invoke([HumanMessage("What is 3 * 4?")])
# response.tool_calls = [{'name': 'multiply', 'args': {'a': 3, 'b': 4}, 'id': 'call_xxx'}]

3.3 工具调用闭环 (ReAct Pattern)

ReAct Pattern = Reasoning + Acting
  1. Reason: Agent 调用 LLM，思考是否需要工具
  2. Act: 如需要，LLM 输出 tool_calls
  3. Observe: ToolNode 执行，返回 ToolMessage
  4. Loop: ToolMessage 追加到 messages，返回 Agent
  5. End: 无 tool_calls 时结束

为什么是循环？

因为一个任务可能需要多次工具调用。例如：搜索 → 解析 → 计算 → 回答

3.4 完整的 Tool Calling 实现

📝 实现要点

1. agent_node: 调用绑定了工具的 LLM
2. tool_node: 解析 tool_calls 并执行
3. should_continue: 条件边路由函数
4. ToolMessage: 必须包含 tool_call_id

注意：ToolMessage 的 tool_call_id 必须与 LLM 返回的 tool_call["id"] 一致，否则 LLM 无法匹配结果。 

from langgraph.graph import MessagesState, StateGraph, START, END
from langchain_core.messages import ToolMessage

tools_by_name = {tool.name: tool for tool in tools}

# 1. agent_node: 调用绑定了工具的 LLM
def agent_node(state: MessagesState):
    return {"messages": [model_with_tools.invoke(state["messages"])]}

# 2. tool_node: 解析 tool_calls 并执行
def tool_node(state: MessagesState):
    results = []
    for tc in state["messages"][-1].tool_calls:
        result = tools_by_name[tc["name"]].invoke(tc["args"])
        # 4. ToolMessage: 必须包含 tool_call_id
        results.append(ToolMessage(content=str(result), tool_call_id=tc["id"]))
    return {"messages": results}

# 3. should_continue: 条件边路由函数
def should_continue(state):
    return "tools" if state["messages"][-1].tool_calls else END

3.5 ToolNode：预构建执行器

LangGraph 提供了开箱即用的 ToolNode，简化工具执行逻辑：


from langgraph.prebuilt import ToolNode

# 一行代码创建工具执行节点
tool_node = ToolNode(tools)

# 添加到图中
workflow.add_node("tools", tool_node)
workflow.add_edge("tools", "agent")
何时自定义 tool_node？需要特殊重试逻辑 / 执行前后钩子 / 结果后处理时

ToolNode 内置能力

• 自动解析: 从 AIMessage.tool_calls 提取调用信息
• 并行执行: 多个工具调用可并行运行
• 错误处理: 工具抛异常时返回错误信息
• 标准输出: 返回符合规范的 ToolMessage

📌 便捷工具：create_react_agent

LangGraph 还提供 create_react_agent(model, tools) 一键创建完整 ReAct Agent，无需手动构建图。

3.6 工具调用最佳实践

DO

• Docstring 清晰描述用途和参数
• 使用类型注解 (int, str, List)
• 工具名称语义化
• 返回结构化数据

DON'T

• 工具功能过于复杂
• 缺少错误处理
• 工具间耦合过深
• 敏感操作无确认

提示：将复杂工具拆分为多个小工具，让 LLM 自己组合调用。

3.7 MCP：模型上下文协议

MCP (Model Context Protocol) 是 Anthropic 提出的开放协议，用于标准化 LLM 与外部工具/数据的交互。

MCP vs 传统 Tool Calling：
• 传统方式：每个应用各自实现工具集成
• MCP 方式：统一协议，工具服务化，即插即用

MCP 核心概念

• MCP Server: 提供工具的服务端
• MCP Client: 调用工具的客户端 (LLM 应用)
• Transport: 通信层 (stdio/SSE)

3.8 LangGraph 集成 MCP Tools

LangGraph 可以通过 langchain-mcp-adapters 无缝集成 MCP 工具：


from langchain_mcp_adapters import MultiServerMCPClient
from langgraph.prebuilt import create_react_agent

# 1. 配置 MCP 服务器
mcp_config = {
    "filesystem": {
        "command": "npx",
        "args": ["-y", "@anthropic/mcp-filesystem"],
    },
    "database": {
        "command": "python",
        "args": ["db_mcp_server.py"],
    }
}

# 2. 创建 MCP 客户端
async with MultiServerMCPClient(mcp_config) as client:
    tools = client.get_tools()  # 获取所有 MCP 工具
                    

    # 3. 创建带 MCP 工具的 Agent
    agent = create_react_agent(
        model="gpt-4o",
        tools=tools  # MCP 工具自动转换为 LangChain 工具
    )
    
    # 4. 调用 Agent
    result = await agent.ainvoke({
        "messages": [{
            "role": "user",
            "content": "读取 /data 目录下的文件列表"
        }]
    })

集成优势

• MCP 工具自动转为 LangChain Tool
• 支持多 Server 同时连接
• 工具发现与调用完全透明

04. 记忆与持久化

Memory, Persistence & Checkpointing

记忆系统架构

4.1 Checkpointer：状态快照

每一步执行后自动保存完整状态。

Checkpointer 是什么？
类似游戏的"存档点"，在图执行的每一步自动保存完整状态。可用于：
• 断点续传（用户离开后继续）
• 错误恢复（从上一个成功点重试）
• 时间旅行（回到任意历史状态）

thread_id

区分不同用户/会话的唯一标识，同一 thread_id 的消息会累积。


from langgraph.checkpoint.memory import MemorySaver

# 创建检查点存储
checkpointer = MemorySaver()

# 编译时注入
app = workflow.compile(checkpointer=checkpointer)

# 运行时传入 thread_id
config = {"configurable": {"thread_id": "user_123"}}
result = app.invoke(inputs, config=config)

4.2 存储后端对比

存储类型 适用场景 持久化 并发支持
MemorySaver 开发、测试、Demo 否 单进程
SqliteSaver 单机部署、轻量应用 是 有限
PostgresSaver 生产环境、多实例 是 高并发 

# PostgreSQL 异步存储
from langgraph.checkpoint.postgres.aio import AsyncPostgresSaver
async with AsyncPostgresSaver.from_conn_string(conn_string) as checkpointer:
    app = workflow.compile(checkpointer=checkpointer)

4.3 跨会话长期记忆 (Store)

除了 Checkpointer 的会话状态，还支持跨会话的长期记忆：

Store vs Checkpointer 区别：
• Checkpointer: 绑定 thread_id，记录单个会话的完整状态历史
• Store: 跨 thread_id，存储用户偏好、知识库等持久数据 

from langgraph.store.memory import InMemoryStore

# 创建长期记忆存储
store = InMemoryStore()

# 编译时同时注入
app = workflow.compile(
    checkpointer=checkpointer,
    store=store
)

# 在节点中访问 Store
def my_node(state, config, store):
    user_prefs = store.get(namespace, key)

4.4 Time Travel (时间旅行)

由于 Checkpointer 保存了每步状态，你可以：

  • 查看历史: 获取每一步的 State 快照
  • Fork: 从历史节点分叉，修改状态后继续
  • Replay: 从任意检查点重新执行

# 查看历史状态列表
for snapshot in app.get_state_history(config):
    print(f"Step {snapshot.step}: {snapshot.values}")

# 从历史节点 Fork
history = list(app.get_state_history(config))
old_state = history[2]  # 获取 Step 2

# 修改状态后继续执行
app.update_state(old_state.config, {"messages": [...]})

调试

回到出错点重新运行

A/B 测试

同一起点尝试不同策略

人工修正

回退并修改 Agent 决策

4.5 Human-in-the-loop (人机协同)


from langgraph.types import interrupt, Command

def approval_node(state):
    # 暂停并向外部暴露信息
    decision = interrupt({
        "question": "确认执行此操作?",
        "details": state["action_details"]
    })
    # 根据人工决策路由
    return Command(
        goto="execute" if decision else "cancel"
    )

# 编译时指定中断点
app = workflow.compile(
    interrupt_before=["approval"])
工作流程：Agent 执行 → 到达 interrupt 节点 → 图暂停，状态保存 → 等待人工响应 → resume 继续执行

05. 高级模式

Multi-Agent Patterns for Production

多智能体架构选型

5.1 Supervisor Pattern (主管模式)

一个 LLM 充当 路由器/调度器，指挥多个 Worker。

  • Supervisor 决定任务分配
  • Workers 专注于各自领域
  • 适合：多角色协作场景
典型场景：研究报告生成
Supervisor → Researcher (搜索) → Writer (撰写) → Reviewer (审核)

5.2 Orchestrator-Worker (编排模式)

Orchestrator 将大任务拆分为子任务：

  1. Orchestrator 生成计划 (子任务列表)
  2. 动态启动 N 个 Worker (Send API)
  3. Workers 并行执行
  4. Synthesizer 合并结果

适用场景

• Map-Reduce: 分析多个文档，汇总结果
• 批量处理: 同时调用多个 API
• 任务数动态: 任务数量运行时才知道
• 效率优先: 子任务无依赖可并行


from langgraph.types import Send

def orchestrator(state):
    """LLM 生成子任务列表"""
    plan = model.invoke(
        f"拆分任务: {state['task']}"
    )
    return {"sections": plan.sections}

def assign_workers(state):
    """Send API: 动态创建并行分支"""
    return [
        Send("worker", {"section": s}) 
        for s in state["sections"]
    ]

graph.add_conditional_edges(
    "orchestrator", assign_workers)

vs Supervisor：Supervisor 逐个调度 Worker，Orchestrator 一次性派发所有子任务并行执行，效率更高。

5.3 Subgraphs (子图嵌套)

将一个完整的图作为另一个图的节点：


# 1. 构建子图
subgraph = StateGraph(State)
subgraph.add_node("search", search_node)
subgraph.add_node("summarize", summarize_node)
sub_app = subgraph.compile()

# 2. 将子图作为节点
parent.add_node("research", sub_app)

# Checkpointer 自动传播到子图

优势

• 模块化：独立开发测试
• 可复用：同一子图用于多处

5.4 Reflection Pattern (反思模式)

通过自我批评迭代提升输出质量。

应用场景：代码生成+自动Review、文章写作+自动润色、方案设计+自动挑战 

def generator(state):
    feedback = state.get("feedback", "")
    prompt = f"任务: {state['task']}\n反馈: {feedback}"
    draft = model.invoke(prompt)
    return {"draft": draft, "iteration": state.get("iteration", 0) + 1}

def reflector(state):
    review = model.invoke(f"评估草稿:\n{state['draft']}")
    return {"feedback": review, "approved": "通过" in review}

def should_continue(state):
    if state["approved"] or state["iteration"] >= 3:
        return END
    return "generator"

graph.add_edge("generator", "reflector")
graph.add_conditional_edges("reflector", should_continue)

06. 最佳实践与总结

Best Practices & Summary

LangGraph 开发最佳实践

架构设计

• 从单 Agent 开始，避免过度设计
• State 字段精简，只存必要数据
• 使用 MessagesState 作为起点
• 子图模块化，便于测试复用

工具设计

• 工具功能单一，名称语义化
• Docstring 清晰描述用途
• 敏感操作加 HITL 确认
• 工具返回结构化数据

记忆管理

• 开发用 MemorySaver，生产用 Postgres
• 合理设置 thread_id 粒度
• 长对话注意 Token 限制
• 跨会话数据用 Store

调试技巧

• 使用 LangSmith 追踪执行
• stream_mode="updates" 观察每步
• get_state_history 回溯问题
• 单元测试每个节点函数

常见陷阱与规避

陷阱 解决方案
无限循环 设置 recursion_limit；条件边确保有退出路径
State 过大导致序列化慢 只存必要字段；大文件存外部引用
工具调用失败 ToolNode 自带重试；关键工具加 try-catch
多节点同时更新冲突 明确定义 Reducer 合并策略
Token 超限 实现消息裁剪策略；使用 Summarizer
