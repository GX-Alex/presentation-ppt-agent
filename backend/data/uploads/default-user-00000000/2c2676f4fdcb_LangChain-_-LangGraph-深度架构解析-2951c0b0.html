StateGraph 状态管理

核心概念

  • State：整个图的共享状态对象
  • Nodes：执行特定任务的函数
  • Edges：定义状态流转规则
  • Reducers：状态合并策略
✓ TypeScript First：完整的类型安全支持，编译时错误检测
from typing import TypedDict, Annotated
from langgraph.graph import StateGraph, START, END

class AgentState(TypedDict):
    messages: list
    next_action: str

graph = StateGraph(AgentState)
graph.add_node("agent", agent_node)
graph.add_edge(START, "agent")

代码迁移示例：Message 格式

❌ 旧版本 (0.x)
from langchain.schema import HumanMessage

# 字符串直接传递
response = chat([HumanMessage("Hello")])

# 混合格式
messages = [
    ("system", "You are a helpful assistant"),
    ("human", "Hi!")
]
response = chat(messages)
✅ 新版本 (1.0)
from langchain_core.messages import HumanMessage

# 必须使用 Message 对象
response = chat.invoke([HumanMessage(content="Hello")])

# 严格的消息类型
from langchain_core.messages import SystemMessage, AIMessage
messages = [
    SystemMessage(content="You are helpful"),
    HumanMessage(content="Hi!"),
    AIMessage(content="Hello!")
]
response = chat.invoke(messages)

注意：1.0 强制要求使用类型化 Message 对象，禁止使用元组或裸字符串

代码迁移示例：Chain 构建

❌ 旧版本 (0.x)
from langchain import OpenAI, PromptTemplate
from langchain.chains import LLMChain

llm = OpenAI(temperature=0)
prompt = PromptTemplate(
    input_variables=["product"],
    template="What is a good name for a {product}?"
)
chain = LLMChain(llm=llm, prompt=prompt)
result = chain.run(product="AI startup")
✅ 新版本 (1.0) - LCEL
from langchain_openai import ChatOpenAI
from langchain_core.prompts import PromptTemplate

llm = ChatOpenAI(temperature=0)
prompt = PromptTemplate.from_template(
    "What is a good name for a {product}?"
)

# 使用 | 操作符构建链
chain = prompt | llm
result = chain.invoke({"product": "AI startup"})

改进：LCEL (LangChain Expression Language) 提供更简洁的链式调用和更好的流式支持

构建 LLM 应用的图结构化编排框架

Graph

整个工作流的容器结构，定义系统的拓扑形态，所有节点和边的元数据存储于此。

  • ●定义工作流拓扑
  • ●管理节点注册
  • ●控制执行流程

State

跨图执行传递的共享状态对象，采用 TypedDict 规范，支持类型提示和状态聚合。

  • ●跨节点状态传递
  • ●Reducer 状态合并
  • ●类型安全保证

Node

执行逻辑的最小单元，通常是 Python 函数，接收当前状态返回状态更新。

  • ●独立执行单元
  • ●状态读写操作
  • ●支持异步/流式

Edge

定义节点间的连接关系，控制条件分支和循环逻辑。

  • ●条件路由
  • ●静态连接
  • ●循环控制

状态持久化核心

StateGraph 是 LangGraph 的核心组件，通过单一共享状态对象协调多个节点的执行。每个节点可以读取状态并返回更新操作，由Reducer决定如何合并这些更新。

state_schema.py
from typing import TypedDict, Annotated
from langgraph.graph import StateGraph, END

class AgentState(TypedDict):
    """Agent 共享状态定义"""
    messages: Annotated[list, add_messages] # 使用Reducer
    context: str # 直接覆盖
    current_node: str # 当前执行节点
    iteration: int # 迭代计数器

Reducer 合并策略

Reducer 是状态更新的核心逻辑函数，决定多个节点返回的值如何合并到共享状态中。默认策略是覆盖更新，也可自定义列表追加或其他复杂逻辑。

“让 Agent 像蜂群一样自主协作，每个 Agent 都是独立的智能体，通过共享上下文和状态进行协作。”

  • 无中央调度器

    每个 Agent 自主决策，支持动态路由和灵活的消息传递

  • Context Sharing

    通过 Shared Context 在 Agent 间传递状态和信息

  • 动态注册机制

    运行时动态注册和注销 Agent，支持弹性扩展

  • Election 机制

    内置 Agent 选举算法，支持故障转移和负载均衡

核心机制：Interrupt 中断点

interrupt_before

在节点执行之前中断，等待人类输入后再决定是否继续执行

适用场景：
  • 执行前的审批确认
  • 敏感操作的二次验证
  • 资源分配的最终确认

interrupt_after

在节点执行之后中断，允许人类审查执行结果后再继续

适用场景：
  • 执行结果的审核
  • 内容生成的确认
  • 数据分析的验证
支持同时使用 interrupt_before 和 interrupt_after，实现完整的事前审批和事后确认流程

graph.add_node("approve", approval_node)

graph.set_entry_point("process")
graph.add_edge("process", "approve")
graph.add_edge("approve", END)

# 编译时启用HIL

    checkpointer=checkpointer,
    interrupt_before=["execute"],  # 执行前中断

# 首次运行
config = {"configurable": {"thread_id": "1"}}

workflow.add_node("execute_critical", execute_critical)
workflow.add_edge("ceo_review", "execute_critical")
workflow.add_edge("execute_critical", END)

# 根据风险级别设置不同中断点
app = workflow.compile(
    checkpointer=checkpointer,
    interrupt_before=["manager_review"],      # 低风险
    interrupt_before=["director_review"],      # 中风险  
    interrupt_before=["ceo_review", "execute_critical"],  # 高风险
)

def route_action(state: SecurityState) -> str:
    """根据风险等级路由到不同审批级别"""
    risk_level = state.get("risk_level", "low")
    
    if risk_level == "low":
        return "manager_review"
    elif risk_level == "medium":
        return "director_review"
    elif risk_level == "high":
        return "ceo_review"
    else:
        return END

最佳实践与注意事项

推荐做法

  • 明确标识中断点，提供清晰的上下文信息
  • 在状态中保存足够的决策依据供审批者参考
  • 设置合理的超时机制，避免流程无限期挂起
  • 记录完整的操作审计日志，满足合规要求
  • 使用独立线程管理审批，避免干扰主流程
  • 提供友好的审批UI，降低审批者学习成本
  • 支持批量审批和委托审批，提升效率

避免事项

  • 中断点过于频繁，导致用户体验下降
  • 审批信息不完整，让审批者无法做出判断
  • 缺少状态持久化，重启后丢失待审批任务
  • 并发控制不当，导致同一任务被多次审批
  • 审批超时处理缺失，产生僵尸流程
  • 过度依赖人工审批，未能发挥AI效率优势
  • 缺少通知机制，审批者不知有新任务

State 设计精简化
TypedDict 只存必要字段，避免将完整文档放入 State；大对象改用 key 引用外部存储，降低 Checkpoint 体积。
⚡
优先使用 Streaming 输出
生产环境统一启用 astream_events，配合前端 SSE 实现实时反馈，大幅提升感知性能与用户体验。
🔄
Retry + Fallback 双保险
LLM 调用必配 with_retry() 与 with_fallbacks()，主模型超时自动切换备用，保障可用性 > 99.5%。
🗄️
持久化选型策略
开发用 InMemory，测试用 SQLite，生产用 PostgreSQL+Redis 分层存储；按 TTL 定期清理过期 Checkpoint 防止膨胀。
🔍
全链路可观测性
接入 LangSmith Tracing，每个 Graph 节点打 metadata 标签；Token 用量、延迟分布、错误率纳入 Prometheus 监控。
⚠️ Anti-Patterns 警告
🚫
Graph 节点过度拆分
每个微操作都建成独立节点，导致状态流转开销剧增、调试困难。建议将逻辑内聚度高的步骤合并，单 Graph ≤ 15 节点为宜。
🚫
同步阻塞 I/O 调用
在 async 节点中使用 requests.get() 等同步 HTTP，会阻塞事件循环。必须使用 httpx.AsyncClient。
🚫
忽略 Token 上下文溢出
不加限制地累积历史消息，超出模型 context window 静默截断导致答案失真。应配置 trim_messages() 动态裁剪。
🚫
Prompt 未版本化管理
将 Prompt 硬编码在业务代码中，模型升级后行为无法对比回滚。应使用 LangSmith Hub 或 Git 管理 Prompt 版本，支持 A/B 测试。
🚫
工具无权限校验
Agent 直接调用文件/数据库工具无任何鉴权，存在严重安全风险。必须添加工具级权限白名单与 Human-in-the-Loop 中断确认机制。

def create_optimized_graph():
    graph = StateGraph(State)
    # 声明并行边
    graph.add_edge("research", "analysis")
    graph.add_edge("research", "summarize")
    graph.add_edge("research", "extract")
    # 使用 Send 机制实现动态并行

        results = parallel_research(state.topic)
        return [Send("process", {"data": r}) for r in results]
    return graph.compile()

LangChain 与 LangGraph 的核心区别是什么？
LangChain 提供组件库和链式调用能力，适合简单场景；LangGraph 采用图结构，支持循环和状态持久化，更适合复杂的多 Agent 系统。
LangChain 1.0 相比 0.x 有哪些重大变化？
主要是 API 稳定性提升、更好的类型提示、统一的 Message 格式、以及 LangGraph 的深度集成。
生产环境推荐使用哪种 Checkpointer？
建议使用 PostgreSQL（适合事务需求）或 Redis（适合高并发场景），开发测试可用 MemoryCheckpointer。

核心术语表

🔗 LangChain

用于构建LLM应用的开源框架，提供Prompt模板、Chain机制、Agent抽象、Memory管理、RAG组件等核心功能。

🔄 LangGraph

基于LangChain构建的图结构编排框架，支持Cycle循环、多Agent协作、状态持久化、人机交互等复杂工作流。

📊 StateGraph

LangGraph中的核心数据结构，通过TypedDict定义状态结构，使用Reducers合并状态更新，支持条件路由与循环。

💾 Checkpointer

状态持久化组件，支持Memory/PostgreSQL/Redis等存储后端，实现图执行状态的保存与恢复，支持多版本历史追踪。

🔍 LCEL

LangChain Expression Language，声明式组合LLM组件的表达式语言，支持|Runable并发、批处理、流式处理、异步操作。

🤖 Multi-Agent

多智能体架构模式，包括Supervisor/Hierarchical/Swarm/Resolver等多种协作模式，实现复杂任务的分布式处理。

核心术语表（续）

🔧 Tool Calling

LLM调用外部工具/函数的能力，通过结构化输出（JSON Schema）定义工具接口，实现LLM与外部系统的集成。

📡 Streaming

流式输出机制，将LLM生成的Token以流式方式返回，提升用户体验，支持实时展示生成进度。

👤 Human-in-the-loop

人机协作模式，在关键决策点暂停执行，等待人类确认或输入，实现AI与人类协同工作流程。

📚 RAG

Retrieval-Augmented Generation，检索增强生成，通过向量数据库检索相关文档，注入LLM上下文提升回答准确性。

🏗️ Conditional Branching

条件路由机制，根据当前状态动态选择下一个执行节点，支持if-else、switch等多种路由逻辑。

⏸️ Interruption

中断机制，允许在图执行过程中暂停，支持状态保存与恢复，常用于长时任务的手动干预场景。
