核心价值：LLM 原生驱动，端到端自动化

将自然语言需求直接转化为专业演示产物，消除人工编排瓶颈。

技术架构：三层微服务实现高效渲染

Backend (FastAPI) + Frontend (Next.js 14) + Native Renderer (PptxGenJS) 协同工作。

编排系统：五大角色协同完成智能编排

Director → Planner → Scheduler → Orchestrator → Reviewer 全链路质量管控。

中间件链：14 层中间件实现复杂任务处理

MemoryCapture、IntentDetection、TokenBudget、LoopDetection 等中间件协同保障任务执行。

核心能力一：Web Deck 演示文稿自动化

核心能力三：Web Sandbox 实时交互

三层微服务架构总览

采用经典的三层微服务架构，各服务职责清晰，通过 Docker Compose 实现一键部署，实现前后端分离与原生渲染解耦。

用户交互层 / Frontend

基于 Next.js 14 + React + TypeScript 构建，包含工作区面板、对话面板与实时预览引擎。通过 Zustand Store 实现状态管理。

核心服务层 / Backend

基于 FastAPI 构建的 AI Agent 运行时引擎。核心组件包括 AgentFactory、MiddlewareChain、AgentRunner，处理流式 LLM 调用 chat_stream。

渲染输出层 / Native Renderer

基于 PptxGenJS 的原生 PPTX 渲染引擎，将结构化数据转换为二进制演示文件，支持多格式内容输出。

通信协议 / Protocol

WebSocket 负责实时双向通信；REST API 提供标准化接口；二进制 PPTX 通过 Base64 编码传输，确保数据类型安全。

AgentRuntime 主循环引擎

AgentRuntime 是 PresentationAgent 的核心执行引擎，通过 AgentFactory 动态组装 AgentContext 与 MiddlewareChain，由 AgentRunner 驱动主循环，实现流式 LLM 调用的全链路控制。

AgentFactory 动态构建

Factory 接收配置参数，动态实例化 AgentContext 与 MiddlewareChain，支持运行时策略调整，实现引擎的可插拔化配置。

AgentContext 上下文管理

Context 封装会话状态、Memory、工具集与运行时元数据，为主循环提供完整的执行上下文，支持跨轮次的上下文累积与压缩。

MiddlewareChain 执行时序

14 层中间件链按序执行：MemoryCapture → AttachmentInjection → ToolError → IntentDetection → TokenBudget → LoopDetection → Checkpoint → Subagent → BriefEnrichment → WebDeckContext → PPTEvent → ... → FinalOutput。每层可修改上下文或终止流程。

AgentRunner 主循环

Runner 循环执行：接收用户输入 → 经过 MiddlewareChain 处理 → 调用 LLM → 流式接收响应 → 更新 Context → 判断是否继续。循环终止条件：输出完成 / 达到 max_turns / 中间件终止。

流式 LLM 调用机制

通过 chat_stream 实现实时流式输出：LLM 分词逐块返回 → Runner 边接收边推送 → 前端实时渲染。Base64 编码的二进制 PPTX 数据亦通过流式传输。

Checkpointer 检查点机制

每轮执行后自动保存 Context 快照，支持断点续传与回滚，确保长时间对话的可靠性与可恢复性。

IntentDetectionMiddleware 解析自然语言意图，智能路由至核心能力模块

意图检测与分发策略机制说明

IntentDetectionMiddleware 是 14 层中间件链中的关键环节。它接收用户自然语言输入，通过 语义理解 + 模式匹配 双引擎解析用户真实意图，然后根据路由策略将请求分发到对应的核心能力模块（WebDeck/DrawIO/WebSandbox/智能文档）进行处理。

意图检测实现机制

  • 语义理解引擎：基于 LLM 进行语义理解，提取用户意图关键词
  • 模式匹配器：预定义意图模式库（创建/编辑/查询/生成等）
  • 上下文感知：结合对话历史和当前上下文进行意图校正
  • 置信度评估：输出意图置信度，低于阈值时触发澄清

路由分发策略

  • 能力映射表：意图类型 → 核心能力模块的映射关系
  • 多意图处理：支持单次请求多意图拆分并发处理
  • 回退机制：无匹配时路由至通用处理模块
  • 动态权重：根据模块可用性和负载动态调整路由

WebDeck 编排系统概述

WebDeck 编排系统通过五大角色的串行协作，实现从需求到演示文稿的完整生成流程。

Director · 总控角色

接收用户需求，统筹全局工作流，管理 AgentContext 生命周期，调度下游角色执行任务。

Planner · 规划角色

分析需求意图，生成演示文稿结构大纲（Slide Outline），输出结构化 Brief 给下游角色。

Scheduler · 调度角色

基于 Lane 机制管理任务队列，实现多页面并行处理，控制系统并发度与资源分配。

Orchestrator · 编排角色

执行具体页面内容生成，调用中间件链处理上下文，生成单页 Slide 内容。

Reviewer · 质检角色

校验生成内容质量，触发重试机制，支持 edit_deck_page 精准编辑与 regenerate 重新生成。

Director：总控角色

编排系统总控入口 · 协调 Planner · Planner · Scheduler · Orchestrator · Reviewer 五大角色协同工作

任务分解与指令下发

Director 接收用户意图后，将复杂任务拆解为原子化指令单元，通过标准化接口分发给 Planner 进行内容规划、Scheduler 进行资源调度、Orchestrator 进行页面编排、Reviewer 进行质量审核。

核心能力： 意图理解 · 任务拆分 · 指令路由

全局状态同步与协调

维护全局执行上下文，统一管理各角色的状态流转。实时监控任务执行进度，处理跨角色依赖关系，确保页面生成、图表绘制、沙盒构建等操作的时序正确性。

核心能力： 状态管理 · 依赖调度 · 进度追踪

异常处理与降级策略

捕获并处理子角色执行异常，实现优雅降级。支持自动重试机制，必要时触发回滚操作，保障系统整体可用性。

Lane 调度核心逻辑

Scheduler 将 DAG 依赖图按拓扑序划分为多个 Lane，同一 Lane 内页面无依赖可并行执行，跨 Lane 需等待前置 Lane 完成。

阶段 组件 作用
依赖建模 DependencyGraph 构建页面间DAG
分层排序 TopoSorter Kahn算法拓扑排序
并行分组 LaneAssigner 分配至同一Lane
执行同步 SyncBarrier 跨Lane同步等待
异常重试 RetryController 失败页面重调度

并行处理策略与优势

通过 Lane 划分实现多页面并行生成，最大化利用 GPU/IO 资源，显著缩短整体生成时间。

资源利用率

同一 Lane 页面共享 GPU 显存，减少上下文切换

容错恢复

单页失败不影响同Lane其他页面，独立重试

负载均衡

动态分配策略避免单点瓶颈

时间优化

10页并行生成耗时约为串行的 30-40%

Kahn 算法简要说明

算法思想：每次选择入度为 0 的节点（无前置依赖），将其加入当前 Lane；移除该节点后更新后续节点入度，重复此过程直至所有节点分配完毕。
PresentationAgent 实现：TopoSorter 对 14 层中间件链的输出页面依赖图执行 Kahn 排序，输出有序 Lane 列表供 LaneAssigner 分配资源。

角色定位

Reviewer 是 WebDeck 编排系统的最终质检节点，在 Orchestrator 完成页面编排后介入，对生成内容进行多维度评估，确保输出符合质量标准并可交付。

质检时机：页面生成后首次质检 → 发现问题触发重试 → 重试后再次质检 → 通过则进入交付流程

质量检查六大维度

内容完整性

检查核心信息是否完整无遗漏，关键数据点是否呈现

逻辑一致性

验证页面间逻辑连贯，层级结构清晰无矛盾

视觉规范性

布局比例、字体大小、间距符合设计规范

格式正确性

PPT/PDF 导出格式正确，无渲染异常

数据准确性

数据引用、统计数值与源文件一致

可读性评估

文字表达清晰度、图表可理解性达标

反馈与建议机制

edit_deck_page

regenerate_deck_page

重新生成指定页面，基于最新上下文和约束条件重新计算布局与内容

retry_failed_deck_pages

批量重试失败页面，自动诊断错误原因并重新执行生成流程

编辑工具典型使用场景

文本内容微调

当页面中某段描述需要更新术语、补充数据或调整措辞时，使用 edit_deck_page 定位目标页面进行精准替换，无需重新生成整份文档

布局结构优化

某些页面首次生成效果不理想（如图表比例失调、信息分布不均），使用 regenerate_deck_page 重新计算布局参数，获得更优呈现效果

错误页面修复

生成过程中因网络超时、API限制等导致部分页面失败，调用 retry_failed_deck_pages 自动重试，系统自动记录失败原因并优化重试策略

批量模板更新

需要统一修改多页的页眉样式、配色方案或字体设置时，批量调用 edit_deck_page 实现全局一致性调整，大幅提升编辑效率

compress_context 压缩机制

针对长对话场景的上下文管理优化机制，通过智能摘要和关键信息提取保持对话连贯性，避免超出模型 Token 限制。

阶段 触发条件
Token 监测 >80% limit
摘要生成 LLM summary
历史压缩 70% reduction
关键保留 tool calls

模型动态切换

AgentFactory 在初始化 AgentContext 时可指定 provider/model 参数，支持运行时动态切换底层模型，实现成本与效果的平衡。

TokenBudgetMiddleware

14层中间件链中的 Token 预算管理组件，实时监控对话 Token 消耗，当接近阈值时触发压缩流程。

上下文连贯性保障

压缩时保留关键信息：用户意图、已确认的工具调用结果、当前任务进度，确保压缩后仍可正确继续对话。
